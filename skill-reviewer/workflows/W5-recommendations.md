# W5 整改方向（从 W3 命中项动态生成）

## 生成规则

1. **扫描 W3 输出**，收集所有命中的检查项编号（如 S1、P3、T3）
2. **查表**：在下方映射表中找到对应行
3. **填充模板**：把方向模板中的占位符替换为该 Skill 的具体文件名
4. **按 W3 优先级分组**：P0 命中 → 第一优先级；P1 → 第二；P2 → 第三

> **关键原则**：W5 的每条建议必须能在 W3 的命中项里找到对应编号，否则不输出。
> **裁判边界**：W5 只指出"需要改什么"和"为什么改"，不给出"具体怎么改"。

---

## 映射表

### S 结构类

| 命中编号 | 整改方向                                                 | 为什么                                      |
| -------- | -------------------------------------------------------- | ------------------------------------------- |
| S1       | `<具体文件>` 中的领域知识需下沉到独立 reference 文件     | 主文档稳定，知识基线可独立更新              |
| S2       | `<具体文件>` 的散文段落需改为结构化标记                  | 无结构化标记会让 Agent 解析失败             |
| S3       | README 与 SKILL.md 的重复描述需删除                      | 流程一改就漂移，同步成本高                  |
| S4       | SKILL.md 需拆分为主文档 + references，确保 ≤ 500 行      | 超出上下文预算，Agent 遗忘率激增            |
| S5       | 需补全 frontmatter 的 `name` 与 `description` 字段       | 无 frontmatter 无法被 Agent Skills 系统触发 |
| S6       | Frontmatter 字段需参考 style-guide 精简                 | 过多字段增加维护成本和漂移风险             |

### C 上下文预算类

| 命中编号 | 整改方向                                              | 为什么                                 |
| -------- | ----------------------------------------------------- | -------------------------------------- |
| C1       | `<具体文件>` 中的通用知识段落需删除                   | Agent 已知通用知识，重复说明浪费 Token |
| C2       | 超出主职责的部分需拆出独立 Skill 或归档               | 范围发散会让触发精度下降               |
| C3       | 超过 50 行的领域细节需移到 `references/`              | 让 Agent 按需加载，节省上下文          |
| C4       | `references/` 文件需加入「何时加载」触发条件          | 无触发条件 Agent 会预加载浪费上下文    |

### I 实现类

| 命中编号 | 整改方向                                            | 为什么                                          |
| -------- | --------------------------------------------------- | ----------------------------------------------- |
| I1       | 文档描述的执行路径需沉淀为真实脚本入口              | 每次执行从文档复制实现会让文档与代码漂移        |
| I2       | 需新增至少一个可独立运行的脚本入口                  | 没有脚本意味着 Skill 只是文档，不是工具         |
| I3       | 需显式声明产物路径                                  | 产物路径不明 Agent 会随意选择，后续步骤无法对接 |

### T 触发与意图类

| 命中编号 | 整改方向                                           | 为什么                            |
| -------- | -------------------------------------------------- | --------------------------------- |
| T1       | description 需改为动词开头并嵌入意图关键词         | 描述需让 Agent 一眼识别触发条件   |
| T2       | description 需加入「不适用于：...」边界声明        | 节省 Context Budget，避免越界触发 |
| T3       | description 需补全技术特征 / 项目环境维度          | 三维触发是 description 质量的核心 |
| T4       | 需拆分多重职责为独立 Skill 或合并到主职责          | 多重职责导致触发竞争              |
| T5       | description 需加强引导性，建议转交 skill-creator 做触发率校准 | 对抗 Agent 的 undertrigger 倾向   |

### M 模式与深度类

| 命中编号 | 整改方向                                                     | 为什么                                       |
| -------- | ------------------------------------------------------------ | -------------------------------------------- |
| M1       | 需新增 Gotchas 段，列出环境特定反直觉事实                    | 没有 Gotchas，Agent 会反复犯同类错误         |
| M2       | 输出格式描述需改为可填空的 Template                          | 散文描述 Agent 难以复刻，模板可精确套用      |
| M3       | 多步工作流需加显式 Checklist                                 | 多步流程无 Checklist 会跳步                  |
| M4       | 需加入 Validation Loop（自检脚本 / 字段校验）                | 让 Agent 在交付前自我确认结果合规            |
| M5       | 破坏性操作需重构为 Plan → Validate → Handoff 三步            | 破坏性操作直接 Execute 风险过高              |
| M6       | 特定示例指令需改写为方法论步骤                                | 避免 Overfitting，让 Skill 处理一类问题      |
| M7       | MUST 类指令需追加「Why」解释                                  | LLM 有 Theory of Mind，解释 Why 比强制更有效 |

### P 脚本设计类

| 命中编号 | 整改方向                                                   | 为什么                                   |
| -------- | ---------------------------------------------------------- | ---------------------------------------- |
| P1       | 一次性命令需锁定版本                                       | 不锁版本会让 Skill 在工具升级后突然失败  |
| P2       | 脚本需加内联依赖声明（PEP 723 / npm:）                     | 自包含依赖让脚本可在任意环境运行         |
| P3       | 交互式调用需改为命令行参数                                 | Agent 运行在非交互 Shell，交互式会挂死   |
| P4       | 脚本需实现 `--help` 自描述                                 | Agent 通过 --help 决定如何调用           |
| P5       | 错误信息需改为可操作格式（出错处 + 预期 + 建议）           | 只报 Error 浪费 Agent 的 turn            |
| P6       | 脚本需分离 stdout（数据）/ stderr（诊断），支持结构化输出  | 自由文本难以被 Agent 解析                |
| P7       | 脚本需加输出截断预案（默认摘要 / --offset）                | 海量输出会被 Agent harness 截断          |

| 命中编号 | 整改方向                                                | 为什么                                |
| -------- | ------------------------------------------------------- | ------------------------------------- |
| V1       | 数据获取步骤后需加入成功判定                             | 没有判定 Agent 无法判断是否进入下一步 |

| V3       | 最终产物处需加落地检查                                   | 不检查产物等于不验收                  |

## 输出格式

```
**第一优先级（W3 命中的 P0 项映射）​**
- <整改方向> —— <为什么>（命中：S1、P3）
- <整改方向> —— <为什么>（命中：M1）

**第二优先级（W3 命中的 P1 项映射）​**
- <整改方向> —— <为什么>（命中：C4、I1）

**第三优先级（W3 命中的 P2 项映射）​**
- <整改方向> —— <为什么>（命中：B1、P7）

**转交建议**
以下整改方向建议转交 skill-creator 执行具体优化：
- <命中编号>：<方向>（skill-creator 可提供数据驱动的触发率校准与迭代优化）
```

每条方向末尾必须标注**命中的 checklist 编号**，让用户能反向追溯到证据。
