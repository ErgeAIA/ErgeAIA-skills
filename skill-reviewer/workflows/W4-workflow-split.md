# W4 拆分需求识别

## 触发条件
仅当 W1 判定为 **中等 / 中等偏复杂 / 复杂** 时执行；轻量级跳过本步骤。

## 目标
识别当前 Skill 中**需要拆分的职责与耦合点**，输出拆分需求清单。具体拆分方案的设计与实施由 skill-creator 执行，裁判不设计目标架构。

---

## 识别原则

### 原则 1：职责单一
一个工作流同时承担 ≥2 个互不相关的职责时，标记为拆分候选。

### 原则 2：耦合信号
以下信号表明需要拆分：
- 主文档同时承担「路由 + 知识库 + 实现」三重职责
- 单个 reference 文件被不同任务路径交叉引用
- 脚本入口承担了预处理 + 核心逻辑 + 后处理

### 原则 3：触发竞争
description 描述了 ≥2 个不同意图，导致触发精度下降时，标记为拆分候选。

---

## 输出格式

| 拆分候选 | 当前耦合的职责 | 拆分理由 | 优先级 | 建议转交 |
| -------- | -------------- | -------- | ------ | -------- |
| `<具体文件/模块>` | 职责A + 职责B | 为什么耦合有害 | P0/P1/P2 | skill-creator |

**填写要求**：
- **当前耦合的职责**：列出当前混在一起的具体职责
- **拆分理由**：说明耦合导致的长期后果（如触发竞争、上下文浪费、维护漂移）
- **优先级**：P0（阻塞使用）、P1（维护风险）、P2（演进风险）
- **建议转交**：具体拆分方案的设计与实施应转交 skill-creator

---

## 规则
- 至少识别 1 个拆分候选（中等及以上复杂度必然存在耦合）
- 每个候选必须引用具体文件名作为证据
- 不设计目标架构，不输出完整工作流表格
- 具体拆分方案由 skill-creator 基于本清单设计
