# W1 复杂度判定

## 契约
本工作流是复杂度档位的**唯一判定者**。
判定标准（六维定义、命中阈值、档位映射、差异化处理）见 `references/rubrics/complexity-rubric.md`，
本工作流仅负责执行流程，**不内嵌标尺定义**。

---

## 执行流程

### Step 1：加载标尺
读取 `references/rubrics/complexity-rubric.md`，确认六维触发条件与当前版本的档位映射。

### Step 2：逐维扫描
对照标尺定义的六维度（阶段性产物 / 执行链阶段 / 运行时依赖 / 扩展面 / 入口数量 / 脚本生态），逐项判定被评审 Skill 是否命中。
每项命中必须落到具体证据（文件名 / frontmatter 字段 / 脚本路径）。

### Step 3：档位映射
按标尺的命中数 → 档位映射表确定档位：
- **0–1 项**：轻量
- **2–3 项**：中等
- **4 项**：中等偏复杂
- **5–6 项**：复杂

### Step 4：扫描渐进式披露成熟度（额外信号）
对照标尺第三节，记录三项观察结果（主文档行数 / references 触发条件 / assets 加载策略）。
这三项**不计入档位**，但作为额外信号传递给 W2（成熟项写优点）与 W3（缺失项写问题）。

### Step 5：决定下游动作

| 档位              | 是否进 W4                    | 是否强制要求 Trace                                          |
| ----------------- | ---------------------------- | ----------------------------------------------------------- |
| 轻量              | 否（显式跳过并在报告中注明） | 否                                                          |
| 中等 / 中等偏复杂 | 是                           | 否（如用户已提供则纳入分析）                                |
| 复杂              | 是                           | **是**（W3 必须包含执行轨迹分析，参见 rubric 差异化处理段） |

---

## 输出格式

> **结论**：<档位>
>
> **判断依据**（六维命中）：
> - [维度名] 具体理由（引用文件名 / 字段 / 路径）
> - [维度名] 具体理由
> - ...
>
> **渐进式披露成熟度**：<成熟 / 有改进空间>
> - 主文档行数：<具体数字> / 5000 Token
> - references 触发条件：<齐 / 缺>
> - assets 加载策略：<按需 / 预加载>

---

## 规则

- 至少给出 3 条判断依据
- 每条必须引用具体文件名 / frontmatter 字段 / 脚本路径作为证据
- 不写"无此维度"占位
- 标尺定义变更**只改** `complexity-rubric.md`，本工作流不动
- 渐进式披露三项观察必须全列，缺一不可
