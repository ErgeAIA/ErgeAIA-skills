# Agent Skill 最佳实践规范

> 加载触发：评审模式进入 W3 且需要溯源官方原则时；或用户明确询问"为什么这条算问题"时。

## 一、起点：源自真实专业知识

不要让 LLM 凭通用训练知识凭空生成 Skill。

### 1.1 从动手任务中提炼
完成一次真实任务后抽取模式，关注：奏效的步骤、你做的纠正、输入输出格式、你提供的上下文。

### 1.2 从既有项目资产中合成
优质素材：内部文档/runbook、API 规格、代码评审评论、版本控制 patch、真实失败案例。通用文章合成的 Skill 会输给项目特定素材合成的 Skill。

### 1.3 用真实执行迭代
跑真实任务 → 把所有结果回灌。阅读执行轨迹（trace）而非仅最终输出。

## 二、上下文预算

### 2.1 加 Agent 缺失的，省 Agent 已知的
对每段内容自问："没有这条，Agent 会做错吗？"答案为否就删。

### 2.2 设计内聚的单元
Skill 粒度像函数：太窄需多个 Skill 共同加载；太宽难以精准触发。

### 2.3 适中详细度
简洁分步 + 工作示例 优于 穷举式文档。

### 2.4 渐进式披露
`SKILL.md` ≤500 行 / 5000 Token。详细资料移到 `references/`，并告诉 Agent **何时**加载：
- ✅ "当 API 返回非 200 时读取 `references/api-errors.md`"
- ❌ "详见 references/"

## 三、控制力度校准

### 3.1 弹性任务给自由
解释"为什么"比给死板指令更有效。

### 3.2 脆弱操作强规定
精确序列 + 禁止偏离。

### 3.3 给默认值不给菜单
挑一个默认推荐，备选项简短带过。

### 3.4 偏好"做法"而非"答案"
教方法论，不给具体实例答案。

## 四、有效指令模式

### 4.1 Gotchas 段（价值密度最高）
列环境特定的反直觉事实：
- 表用软删除，必须带 `WHERE deleted_at IS NULL`
- 同一 ID 在不同服务字段名不同（`user_id` / `uid` / `accountId`）
- `/health` 总返回 200，要用 `/ready` 检查全栈

**迭代规则**：每当 Agent 犯一个你必须纠正的错，就加进 Gotchas。

### 4.2 输出格式模板
给具体结构模板而非散文描述。

### 4.3 多步工作流 Checklist
显式列出步骤，标注依赖与验证关卡。

### 4.4 Validation Loop
做 → 跑校验 → 修复 → 重跑直到通过。

### 4.5 Plan-Validate-Handoff
破坏性/批量操作：生成结构化计划 → 对照真实源验证 → 转交执行。错误信息要具体到字段名级别。

### 4.6 打包可复用脚本的信号
若 Agent 每次都重新发明同一段逻辑（建图、解析、校验），写一个测试过的脚本打包到 `scripts/`。

## 五、脚本设计

### 5.1 一次性命令
固定版本（`npx eslint@9.0.0`）。复杂命令转脚本。

### 5.2 自包含脚本
- Python：PEP 723 / `uv run`
- Deno：`npm:` / `jsr:` 导入
- Bun：导入路径含版本
- Ruby：`bundler/inline`

### 5.3 Agent 友好硬规则
- **严禁交互式提示**：Agent 在非交互 Shell，TTY 阻塞会挂死
- **用 `--help` 自描述**：简洁，会进入 Agent 上下文
- **错误信息可操作**：说错处 + 预期 + 建议
- **结构化输出**：JSON / CSV / TSV，stdout 出数据，stderr 出诊断
- **幂等性**：Agent 可能重试
- **Dry-run 支持**：破坏性操作必备
- **退出码区分失败类型**：在 `--help` 中文档化
- **输出大小预案**：默认摘要 + `--offset` / `--output`

## 六、触发优化

`description` 字段决定 Skill 何时激活：
- 动词开头
- 覆盖触发短语多种说法
- 声明不处理的边界
- 维护正面/负面测试集做回归

## 七、评估

- 测试用例 + 可机器判定的断言 + 评分
- 读执行轨迹，发现：指令模糊 / 指令不适用 / 选项太多无默认
- 单次 execute-then-revise 就有显著提升

## 八、工程原则（源自 Karpathy 技能规范）

> 来源：[forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)
> 解决 LLM 在编程任务中的四个典型问题：错误假设、不管理困惑、过度复杂化、乱改代码。

### 8.1 Think Before Coding（动手前先思考）
- 明确陈述假设——不确定就问，不要猜
- 存在歧义时，呈现多个解释供选择
- 有更简单的方案时，说出来
- 困惑时停下来，明确说出不清楚的地方

**审查映射**：当 Skill 的指令隐含假设但未显式声明时，命中 M7（MUST 类指令缺 Why 解释）。

### 8.2 Simplicity First（简单优先）
- 不加需求之外的功能
- 不为单次使用的代码创建抽象
- 不添加没被要求的"灵活性"或"可配置性"
- 如果 200 行能写成 50 行，就重写

**审查映射**：当 Skill 包含未被要求的灵活性或过度抽象时，命中 C2（范围发散）或 S4（主文档过长）。

### 8.3 Surgical Changes（精准改动）
- 只改必须改的
- 不要"改进"相邻代码、注释或格式
- 匹配现有风格，即使你会有不同做法
- 如果发现无关的死代码，提及但不要删除

**审查映射**：当 Skill 的修改指令可能引发副作用时，命中 M5（破坏性操作缺 P-V-H 三步）。

### 8.4 Goal-Driven Execution（目标驱动执行）
- 将命令式任务转化为可验证的目标
- 建立验证循环，让 Agent 能独立循环验证
- 成功标准是强声明（"确保 X 测试通过"）而非弱声明（"让它工作"）

**审查映射**：当 Skill 缺少可验证的完成标准时，命中 V1-V3（验证闭环缺失）。
