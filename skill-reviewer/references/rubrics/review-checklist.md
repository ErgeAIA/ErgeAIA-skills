# Skill 评审检查清单（裁判师版）

> 48 项检查点，9 大维度。每项含「编号 / 检查内容 / 优先级 / 对应工作流」四元组。
> 评审时由 W2 与 W3 共同消费本表：命中即落优点（W2 视角）或落问题（W3 视角）。

---

## 使用契约

| 角色                    | 责任                                           |
| ----------------------- | ---------------------------------------------- |
| **本文件 (checklist)​** | 提供判定标准与阈值，**不直接生成评审段落**     |
| **W2**                  | 扫描本表，挑命中强项落入「主要优点」段         |
| **W3**                  | 扫描本表，挑未命中项按优先级落入「主要问题」段 |
| **W7**                  | 仅对 T 系列做深度审计，结果回写 W3 的 P1 子项  |
| **intent-calib**        | 为 T 系列提供示例库与触发哲学，不抢判定权      |

## 字段归属说明

### 官方规范字段（specs/spec-zh.md）

| 字段 | 类型 | 位置 |
|------|------|------|
| `name` | 必需 | 顶层 |
| `description` | 必需 | 顶层 |
| `license` | 可选 | 顶层 |
| `compatibility` | 可选 | 顶层 |
| `metadata` | 可选 | 顶层 |
| `allowed-tools` | 可选 | 顶层 |

### 项目扩展字段（放入 metadata 内部）

| 字段 | 说明 |
|------|------|
| `tags` | 标签列表 |
| `origin` | 技能来源 |
| `repo` | 仓库地址 |
| `updated` | 更新时间 |

> **注意**：`triggers` 不是官方字段，触发信息应直接写入 `description` 中。

### B3 触发条件说明

- 官方规范中 `description` 是主要触发机制，无独立 `triggers` 字段
- 评审时应检查：
  1. `description` 是否同时描述「做什么」与「何时使用」（官方要求）
  2. `description` 是否包含帮助 Agent 识别相关任务的关键词
  3. `description` 是否声明不处理的边界场景（最佳实践，非官方硬限）

> **注意**：官方 `spec.md` 仅定义 6 个 frontmatter 字段（name, description, license, compatibility, metadata, allowed-tools）。`triggers` 不是官方字段，评审时不应将其作为合规要求。

## 检查项总表

### O 起源 (Origin)

| 编号 | 检查内容                                       | 优先级 | 对应工作流 |
| ---- | ---------------------------------------------- | ------ | ---------- |
| O1   | 是否源自真实任务提炼或项目素材合成             | P0     | W2 / W3    |
| O1a  | metadata 是否包含 `origin` 字段                | P2     | W3         |
| O1b  | changelog 是否独立为 VERSION.md（非内嵌）      | P2     | W3         |
| O2   | 是否经过至少一轮 execute-then-revise 迭代      | P2     | W2 / W3    |
| O3   | 是否阅读过执行轨迹 (Trace) 识别 Agent 试错信号 | P2     | W2 / W3    |

### S 结构 (Structure)

| 编号 | 检查内容                                          | 优先级 | 对应工作流      |
| ---- | ------------------------------------------------- | ------ | --------------- |
| S1   | 主文档是否只承担路由职责（不装大段代码 / 知识）   | P0     | W2 / W3         |
| S2   | 主文档是否使用结构化标记（H2 / 列表 / Checklist） | P0     | W3              |
| S3   | README 与 SKILL.md 是否职责分离                   | P1     | W3              |
| S4   | SKILL.md 是否 ≤ 500 行 / 5000 Token               | P0     | V0（硬限） / W3 |
| S5   | 是否含合规 frontmatter (name, description)        | P0     | V0（硬限） / W3 |
| S6   | Frontmatter 字段是否过于复杂（参考 style-guide）   | P2     | W3（软性建议）  |

### C 上下文预算 (Context)

| 编号 | 检查内容                                  | 优先级 | 对应工作流 |
| ---- | ----------------------------------------- | ------ | ---------- |
| C1   | 是否删除了 Agent 已知的通用知识           | P1     | W3         |
| C2   | Skill 范围是否内聚（单一职责）            | P1     | W3         |
| C3   | 详细资料是否下沉到 references/ 或 assets/ | P1     | W2 / W3    |
| C4   | 参考文件是否有显式"何时加载"的触发指引    | P1     | W3         |

### I 实现 (Implementation)

| 编号 | 检查内容                                         | 优先级 | 对应工作流 |
| ---- | ------------------------------------------------ | ------ | ---------- |
| I1   | 文档描述的每条执行路径，仓库里是否有真实脚本入口 | P1     | W3         |
| I2   | 是否存在至少一个可独立运行的脚本                 | P1     | W2 / W3    |
| I3   | 是否有产物路径 (Output Path) 的明确约定          | P2     | W3         |

### T 触发与意图 (Trigger & Intent)

| 编号 | 检查内容                                     | 优先级 | 对应工作流          |
| ---- | -------------------------------------------- | ------ | ------------------- |
| T1   | description 动词开头、含意图关键词           | P0     | **W7（唯一判定）​** |
| T2   | description 声明不处理的边界场景             | P1     | **W7（唯一判定）​** |
| T3   | 三维触发：意图 + 技术特征 + 项目环境         | P0     | **W7（唯一判定）​** |
| T4   | 意图内聚：是否存在不相关的多重职责需拆分     | P2     | **W7（唯一判定）​** |
| T5   | 触发倾向：描述是否足够明确且具引导性 (Pushy) | P2     | **W7（唯一判定）​** |

> **T 系列归属说明**：W3 不直接判 T 系列，统一委托给 W7 深度审计，W7 输出回写 W3 的 P1 子项。

### M 模式与深度 (Advanced Patterns)

| 编号 | 检查内容                                      | 优先级 | 对应工作流 |
| ---- | --------------------------------------------- | ------ | ---------- |
| M1   | 是否有 Gotchas 段且条目是具体纠正             | P0     | W2 / W3    |
| M2   | 输出格式是否有模板而非散文描述                | P1     | W2 / W3    |
| M3   | 多步工作流是否有显式 Checklist                | P1     | W3         |
| M4   | 是否有 Validation Loop 让 Agent 自检          | P1     | W2 / W3    |
| M5   | 破坏性操作是否采用 Plan-Validate-Handoff 模式 | P1     | W2 / W3    |
| M6   | 做法优于答案：是否教方法论而非死指令          | P2     | W2 / W3    |
| M7   | 解释 Why：关键指令是否有逻辑支撑而非单纯 MUST | P2     | W2 / W3    |

### P 脚本设计 (Agentic Scripts)

| 编号 | 检查内容                                           | 优先级 | 对应工作流 |
| ---- | -------------------------------------------------- | ------ | ---------- |
| P1   | 一次性命令是否固定版本 (npx tool@version)          | P2     | W3         |
| P2   | 脚本是否声明内联依赖 (PEP 723 / npm:)              | P1     | W3         |
| P3   | 严禁交互：是否完全无 input() / TTY 阻塞            | P0     | W3         |
| P4   | 是否实现 --help 且简洁自描述                       | P1     | W3         |
| P5   | 错误信息是否可操作（出错处 + 预期 + 建议）         | P1     | W3         |
| P6   | 是否结构化输出且分离 stdout / stderr               | P1     | W3         |
| P7   | 是否考虑输出截断（默认摘要 / --offset / --output） | P2     | W3         |

### V 验证闭环 (Verification)

| 编号 | 检查内容                          | 优先级 | 对应工作流 |
| ---- | --------------------------------- | ------ | ---------- |
| V1   | 数据获取 / 处理是否有成功判定标准 | P1     | W3         |
| V2   | 数据对象是否有字段完整性检查      | P1     | W3         |
| V3   | 最终产物是否有落地检查            | P1     | W3         |
| V4   | 是否有评估测试集（正面集 + 负面集）用于回归验证 | P2 | W3 |
| V5   | 评估断言是否可机器判定（非主观评分） | P2 | W3 |
| V6   | 评审报告 W5 每条建议是否包含 checklist 编号标注 | P1 | W3 |
| V7   | 术语一致性：是否运行 --consistency 检查旧术语残留 | P2 | W3 |

### B 边界 (Boundary)

| 编号 | 检查内容                                   | 优先级 | 对应工作流 |
| ---- | ------------------------------------------ | ------ | ---------- |
| B1   | 是否声明运行时要求 (Python 版本、系统依赖) | P2     | W3         |
| B2   | 非执行性材料（复盘 / 故事）是否被显式标注  | P2     | W3         |
| B3   | 触发条件是否清晰可判定，description 是否包含触发关键词 | P0     | W3         |
| B4   | 非目标 (Non-Goals) 是否被显式列出          | P2     | W3         |

## 阈值

- 评审时**逐项扫描**，命中即由 W2/W3/W7 落成段落
- 未命中**不写占位**
- 每条结论**必须**给出至少一个文件名作为证据
- W6 工程化档位以**未命中数**为依据：
  - ≤ 5 项未命中 → 稳定可复用
  - 6 – 12 → 半产品化
  - 13 – 20 → 执行说明
  - ≥ 21 → 想法阶段

## 维度映射表

> 本表建立 9 维度检查清单与外部规范体系的对应关系，确保审查标准一致性。

### 与 Karpathy 四原则的映射

| Karpathy 原则 | 对应检查项 | 映射说明 |
|---------------|-----------|---------|
| Think Before Coding | M7, O2, O3 | 假设显式化、执行迭代、轨迹阅读 |
| Simplicity First | C2, S4, C1 | 范围内聚、主文档精简、删除通用知识 |
| Surgical Changes | M5, S1, S3 | 破坏性操作 P-V-H、路由职责分离、README 分离 |
| Goal-Driven Execution | V1, V2, V3, M4 | 成功判定、字段检查、落地检查、验证闭环 |

### 与 5 阶段审查流程的映射

| 5 阶段 | 对应检查项 | 映射说明 |
|--------|-----------|---------|
| 1. 结构审查 | S1-S6, C1-C4 | 结构与上下文预算 |
| 2. 内容审查 | O1-O3, M1-M7 | 起源与模式深度 |
| 3. 触发审查 | T1-T5, B3 | 触发与意图 |
| 4. 脚本审查 | P1-P7, I1-I3 | 实现与脚本设计 |
| 5. 验证审查 | V1-V7, B1-B4 | 验证闭环与边界 |

### 优先级统一标准

| 级别 | 定义 | 行动 |
|------|------|------|
| **P0** | 阻塞使用：Agent 挂死、触发失败、合规不通过 | 必须修复 |
| **P1** | 维护风险：知识过期、职责漂移、可维护性下降 | 应当修复 |
| **P2** | 演进风险：环境绑定、缺少自检、可扩展性不足 | 建议修复 |
