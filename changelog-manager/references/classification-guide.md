# 变更分类详细指南

## 标准分类

### Added（新增）
用于所有新添加的功能、文件、模块、配置等。

**适用场景：**
- 新功能实现
- 新增 API 端点
- 新增配置选项
- 新增文件或模块
- 新增依赖库
- 新增文档

**示例：**
```
- 用户注册与登录功能
- 支持 GitHub OAuth 第三方登录
- 新增 `--config` 命令行参数
- 添加单元测试覆盖核心模块
```

**边界案例：**
- 重构后"新增"的内部模块 → 如果对外部用户不可见，考虑 `Changed`
- 重新实现已移除的功能 → `Added`（因为对用户来说是新的）

---

### Changed（变更）
用于对现有功能的修改，但不涉及新增或移除。

**适用场景：**
- 功能行为变更
- API 接口变更（非破坏性）
- 配置默认值变更
- 依赖库升级
- 性能优化
- 算法改进

**示例：**
```
- 默认超时时间从 30s 调整为 60s
- 优化数据库查询性能，响应时间减少 50%
- 升级 React 从 17 到 18
- 搜索结果排序逻辑调整
```

**边界案例：**
- 破坏性变更（Breaking Change）→ `Changed`，但应在条目中明确标注"⚠️ BREAKING"
- Bug 修复 → `Fixed`，不是 `Changed`

---

### Deprecated（弃用）
用于标记即将在未来版本移除的功能。

**适用场景：**
- 即将移除的 API
- 即将替换的配置项
- 即将废弃的功能模块
- 即将更改的接口签名

**示例：**
```
- `User.login()` 方法将在 v2.0 中移除，请使用 `User.authenticate()`
- `config.yaml` 中的 `legacy_mode` 选项将在下个版本废弃
```

**最佳实践：**
- 必须说明替代方案
- 必须说明预计移除的版本
- 给用户足够的迁移时间

---

### Removed（移除）
用于已经移除的功能。

**适用场景：**
- 已删除的功能
- 已删除的 API
- 已删除的配置选项
- 已删除的文件或模块

**示例：**
```
- 移除 v1.x 旧版 API 端点 `/api/v1/users`
- 移除对 Python 3.6 的支持
- 删除未使用的 `utils/legacy.js` 模块
```

**最佳实践：**
- 对应的 `Deprecated` 条目应在此之前的版本中已存在
- 说明移除原因和替代方案

---

### Fixed（修复）
用于 Bug 修复。

**适用场景：**
- 功能异常修复
- 崩溃修复
- 数据错误修复
- 显示问题修复
- 兼容性问题修复

**示例：**
```
- 修复用户上传头像时偶尔失败的问题
- 修复 Safari 浏览器下页面布局错位
- 修复并发请求时数据竞争导致的死锁
- 修复日期选择器在跨时区时显示错误
```

**边界案例：**
- 安全漏洞修复 → `Security`，不是 `Fixed`
- 功能调整导致的"修复" → 如果本质是行为变更，用 `Changed`

---

### Security（安全）
用于安全性相关的改进。

**适用场景：**
- 安全漏洞修复
- 依赖库安全升级
- 新增安全特性
- 权限控制改进
- 数据加密增强

**示例：**
```
- 修复 XSS 跨站脚本攻击漏洞（CVE-2024-XXXX）
- 升级 lodash 至 4.17.21 修复原型污染漏洞
- 新增 API 请求速率限制
- 用户密码存储改用 bcrypt 加密
```

**最佳实践：**
- 引用 CVE 编号（如有）
- 说明影响范围和严重程度

---

## 分类决策树

```
这个变更是...
├─ 全新的东西？ → Added
├─ 不再存在了？
│  ├─ 之前已标记弃用？ → Removed
│  └─ 之前未标记？ → Removed（建议下次先 Deprecated）
├─ 修复了错误？
│  ├─ 与安全相关？ → Security
│  └─ 与安全无关？ → Fixed
├─ 标记即将移除？ → Deprecated
└─ 修改了现有行为？ → Changed
    └─ 是否破坏向后兼容？ → 在条目中标注 ⚠️ BREAKING
```

## 多分类条目

当一个变更涉及多个分类时：
- **选择最显著的分类**作为主要分类
- 在条目描述中补充说明其他方面

**示例：**
```
### Fixed
- 修复登录超时问题，同时优化了错误提示文案（#123）
```

## 不需要记录的变更

以下变更通常不需要记录在更新日志中：
- 代码格式化（Prettier、ESLint 自动修复）
- 注释更新
- 变量重命名（不影响公共 API）
- 内部重构（不影响外部行为）
- CI/CD 配置调整
- .gitignore 更新
